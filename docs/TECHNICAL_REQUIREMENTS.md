# イチゲキーン - 技術要件定義書

## 1. プロジェクト概要 (Project Overview)

### 目的
この技術要件定義書は、PyxelとPythonを使用して開発されるレトロ格闘ゲーム「イチゲキーン」の技術的な側面、要件、および開発プロセスを明確にすることを目的とします。これにより、開発チーム内での認識共有を促進し、効率的な開発と品質確保を目指します。

### 対象ゲーム
「イチゲキーン」は、一撃必殺の緊張感を特徴とする2D対戦型格闘ゲームです。プレイヤーはキャラクターを操作し、相手の隙を突いて攻撃を当てることで勝利を目指します。パリィ（受け流し）システムや、NPC対戦モード、攻撃失敗ペナルティなどの要素が盛り込まれています。

## 2. コア技術 (Core Technologies)

### **ゲームエンジン (Game Engine): Pyxel**
*   **バージョン:** 最新の安定版を推奨します。開発開始前にバージョンを固定し、チームメンバー間で統一します。
*   **初期設定:**
    *   画面サイズ: `256x192` ピクセル。
    *   FPS: `30` FPSをターゲットとします。（GAME_SPEC.mdの記述に基づく。必要に応じて60FPSも検討）
    *   タイトル: 「イチゲキーン」
*   **ゲームループ:**
    *   `update()`: ゲームの状態更新、入力処理、物理演算、衝突判定、AIロジックなど、フレームごとのロジック処理を担当します。
    *   `draw()`: `update()` の結果に基づき、画面に表示される全ての要素（キャラクター、背景、UIなど）の描画処理を担当します。`pyxel.cls()` で画面をクリア後、各要素を描画します。
*   **入力処理:**
    *   `pyxel.btnp(key)`: 攻撃、パリィ、UI決定など、単発のアクション入力の検出に使用します。
    *   `pyxel.btn(key)`: キャラクターの移動など、キーが押され続けている間の継続的な入力検出に使用します。
*   **リソース管理:**
    *   `.pyxres` ファイル: ゲーム内で使用する画像リソース（タイルマップ用タイルセット、キャラクタースプライトシート、エフェクトスプライト）および音声データ（サウンドエフェクト、BGMのパターン）は、Pyxel Editorを使用して単一の `.pyxres` ファイルにまとめて管理します。これにより、アセットの読み込みと配布が簡略化されます。

### **プログラミング言語 (Programming Language): Python**
*   **バージョン:** Python 3.9 以降を推奨します。開発環境の統一のため、具体的なマイナーバージョンまで指定することが望ましいです (例: Python 3.10.x)。
*   **型ヒント:** コードの可読性、保守性の向上、および静的解析ツールによる早期のバグ発見を目的として、原則として型ヒントを使用します。

## 3. グラフィック (Graphics)

### **タイルマップ (Tilemaps - Pyxel):**
*   **ステージ背景:** 各対戦ステージの背景描画、およびプレイヤーが移動できない境界領域の設定には、Pyxelのタイルマップ機能を利用します。
*   **タイルセット:** ステージ背景を構成するタイルパターンは、`.pyxres` ファイル内のタイルマップエディタで定義・管理します。

### **スプライト (Sprites - Pyxel):**
*   **キャラクター:** プレイヤーキャラクターおよびNPCキャラクターのグラフィックは、`.pyxres` ファイル内のイメージバンクにスプライトシートとして格納し、`pyxel.blt()` 命令を使用して画面に描画します。
*   **アニメーション:**
    *   キャラクターの各状態（アイドル、移動、攻撃、パリィ、被ダメージ、勝利/敗北ポーズなど）に対応するアニメーションは、スプライトシート内の連続するフレームを一定間隔で切り替えることで表現します。
    *   アニメーション管理のため、各キャラクターオブジェクト（または専用のアニメーション管理クラス）が現在のアニメーション状態とフレーム番号を保持し、`update()` 内で適切に更新する仕組みを実装します。
*   **エフェクト:** 攻撃時の斬撃エフェクト、ヒット時のインパクトエフェクト、パリィ成功時のエフェクトなども、個別のスプライトまたはスプライトシートとして `.pyxres` ファイルで管理し、必要に応じて描画します。

## 4. サウンド (Sound)

### **サウンドエフェクト (Sound Effects - Pyxel):**
*   **定義:** 攻撃音、剣戟音、パリィ成功音、被ダメージ音、キャラクターの移動音、UIの決定音・キャンセル音などのサウンドエフェクトは、Pyxel Editorのサウンドエディタを使用して作成し、`.pyxres` ファイル内のサウンドバンクに格納します。
*   **再生トリガー:** 各ゲームイベント（例: プレイヤーの攻撃入力時、攻撃ヒット時、パリィ成功時、UI操作時など）に応じて、`pyxel.play(channel, sound_index)` 関数を呼び出し、適切なサウンドエフェクトを再生します。チャンネルの使い分け（例: キャラクターアクション用、UI用など）も考慮します。

### **BGM (Background Music - Pyxel):**
*   **定義:** タイトル画面用、ゲームプレイ中用などのBGMは、Pyxel Editorのミュージックエディタを使用してパターンを作成し、`.pyxres` ファイル内のミュージックバンクに格納します。
*   **再生制御:** 特定のゲーム状態に遷移した際（例: タイトル画面表示時、ゲームプレイ開始時）に `pyxel.playm(music_index, loop=True)` 関数を呼び出し、BGMをループ再生します。状態遷移時には適宜BGMを停止または変更します。

## 5. ゲームロジック (Game Logic - Python)

### **状態管理 (State Management):**
*   **ゲーム状態:** ゲーム全体の流れを制御するため、以下の主要な状態を定義し、これらを管理する仕組みを実装します（例: 有限ステートマシン、専用の管理クラス）。
    *   `STATE_TITLE`: タイトル画面、モード選択
    *   `STATE_CHARACTER_SELECT`: キャラクター選択画面（将来的な拡張用）
    *   `STATE_GAMEPLAY`: 対戦中のメインゲームループ
    *   `STATE_ROUND_OVER`: ラウンド終了時（勝者表示、次のラウンドへの移行準備）
    *   `STATE_GAME_OVER`: 試合終了時（最終的な勝者表示、タイトルへの復帰など）
    *   `STATE_PAUSE`: 一時停止画面（オプション）
*   **プレイヤー状態:** 各プレイヤーキャラクターが取りうる状態を定義し管理します。これにより、アニメーションの切り替えや行動制約などを制御します。
    *   例: `IDLE`, `MOVING`, `ATTACKING_STARTUP`, `ATTACKING_ACTIVE`, `ATTACKING_RECOVERY`, `PARRYING`, `HIT_STUN`, `BLOCK_STUN` (パリィ失敗時など), `VICTORY`, `DEFEAT`。

### **物理・衝突判定 (Physics & Collision Detection):**
*   **移動ロジック:** プレイヤーキャラクターの左右移動、およびオプションのショートダッシュ機能の実装。画面境界やステージ内の障害物との衝突を考慮し、移動範囲を制限します。
*   **攻撃範囲:** 各攻撃アクションに対して、攻撃判定が発生する領域（ヒットボックス）を矩形で定義します。ヒットボックスの位置やサイズは、攻撃アニメーションのフレームに応じて変化する場合があります。
*   **衝突判定:**
    *   攻撃ヒットボックスと、相手キャラクターの喰らい判定用矩形（ハートボックス）との間で、AABB (Axis-Aligned Bounding Box) 方式による衝突判定を毎フレーム行います。
    *   ハートボックスはキャラクターの中心的な領域をカバーし、複数設定することも検討します（例：頭部、胴体）。
*   **パリィ判定:** プレイヤーがパリィアクション中に相手の攻撃ヒットボックスと接触した場合、パリィ成功と判定します。成功時は相手の攻撃を無効化し、相手を一定時間無防備状態（硬直）にするロジックを実装します。

### **NPC AI:**
*   **基本思考ルーチン:** NPC対戦モードにおけるNPCキャラクターの行動パターンを実装します。
    *   移動: プレイヤーとの距離に応じて接近または離脱する基本的な移動ロジック。
    *   攻撃: 一定の条件下（例: プレイヤーが特定範囲内にいる、プレイヤーが隙を見せたなど）で攻撃を繰り出す。
    *   パリィ: プレイヤーの攻撃に対して、一定の確率または特定のタイミングでパリィを試みる。
*   **難易度設定:** 将来的な拡張として、NPCの反応速度、攻撃頻度、パリィ精度などを変更することで、複数の難易度レベル（例: 初級、中級、上級）を提供できるように設計を考慮します。

### **特殊ルール実装 (Special Rules Implementation):**
*   **攻撃失敗ペナルティ:**
    *   各プレイヤーの連続攻撃失敗回数をカウントする変数を管理します。
    *   攻撃が相手にヒットせず、かつパリィもされなかった場合に失敗とカウントします。
    *   連続失敗回数が3回に達した場合、該当プレイヤーキャラクターの移動速度を一定時間（例: 2秒間）、一定割合（例: 50%に低下）で減少させるペナルティを適用します。
    *   ペナルティ適用中に攻撃がヒットするか、一定時間が経過するとペナルティを解除し、移動速度を元に戻します。

## 6. ユーザーインターフェース (User Interface - Pyxel)

*   **画面描画:** Pyxelの描画コマンドを使用して、各ゲーム状態に応じたUI要素を画面に表示します。
    *   テキスト表示: `pyxel.text(x, y, string, color)` を使用し、スコア、ラウンド数、ゲームモード選択肢、メッセージなどを表示します。
    *   図形描画: `pyxel.rect()`, `pyxel.circ()`, `pyxel.line()` などを使用して、選択肢のカーソル、ゲージ（もしあれば）、区切り線などを描画します。
*   **情報表示:**
    *   スコア: 各プレイヤーが獲得したラウンド数を表示します（例: 星やアイコンで表現）。
    *   ラウンド開始/終了: 「Round 1」「Fight!」「Player 1 Wins」「Draw Game」などのテキストを適切なタイミングで表示します。
    *   勝敗結果: ラウンド終了時およびゲーム終了時に、勝者または引き分けを明確に表示します。
*   **メニュー操作:**
    *   タイトル画面: 「2P対戦」「NPC対戦」などのモード選択肢を表示し、プレイヤーの入力（例: 上下キーで選択、攻撃/パリィキーで決定）に応じて状態を遷移させます。
    *   キャラクター選択画面（オプション）: 選択可能なキャラクターを一覧表示し、各プレイヤーがキャラクターを選択できるようにします。
    *   ポーズ画面（オプション）: 「再開」「タイトルへ戻る」などの選択肢を表示します。

## 7. 開発プロセスとツール (Development Process & Tools)

### **テスト (Testing): unittest**
*   **テスト対象:** ゲームのコアロジック（状態遷移の正確性、衝突判定ロジック、スコア計算、特殊ルールの適用条件など）を中心に、Python標準ライブラリの `unittest` モジュールを使用したユニットテストを作成します。
*   **カバレッジ目標:** `coverage.py` ツールを使用し、テストカバレッジを計測します。目標カバレッジ（例: 80%）を設定し、CI/CDプロセスでレポートを確認できるようにします。
*   **テスト構成:** 機能ごと、または重要なクラスごとにテストファイル（例: `test_collision.py`, `test_state_machine.py`）を分離し、体系的に管理します。

### **バージョン管理 (Version Control): Git**
*   **ブランチ戦略:** GitflowまたはGitHub Flowに基づいたブランチ戦略を採用します。
    *   `main`: リリース可能な安定バージョン。
    *   `develop` (または `dev`): 現在の開発が進むメインライン。
    *   `feature/xxx`: 個別の機能開発やバグ修正を行うためのトピックブランチ。
*   **コミット規約:** [Conventional Commits](https://www.conventionalcommits.org/) の規約を導入し、コミットメッセージの接頭辞（例: `feat:`, `fix:`, `docs:`, `style:`, `refactor:`, `test:`, `chore:`）によって変更内容を明確にします。

### **ドキュメンテーション (Documentation): Markdown & Sphinx**
*   **基本ドキュメント:** `GAME_SPEC.md`（ゲームデザインドキュメント）、`TECHNICAL_REQUIREMENTS.md`（本ドキュメント）はMarkdown形式で `docs` ディレクトリ以下に管理します。
*   **コードコメント:** Pythonのdocstring（PEP 257準拠）をクラス、メソッド、関数に適切に記述し、コードの意図と使用方法を明確にします。
*   **高度なドキュメント生成 (Sphinx):**
    *   `docs/source` ディレクトリにreStructuredText (`.rst`) またはMarkdown (`.md`、`myst_parser`を使用) 形式でドキュメントファイルを作成します。
    *   Sphinxを使用してHTMLドキュメントを生成し、APIリファレンス（docstringから自動生成）、詳細な設計解説、チュートリアルなどを提供します。
    *   `docs/source/conf.py` でSphinxの拡張機能（例: `sphinx.ext.autodoc`, `sphinx.ext.napoleon`, `myst_parser`）やテーマを設定します。

### **コードスタイルとフォーマット (Code Style & Formatting): PEP 8 & Ruff**
*   **規約準拠:** Pythonの公式スタイルガイドであるPEP 8に準拠したコード記述を徹底します。
*   **自動フォーマット:** [Ruff](https://github.com/astral-sh/ruff) のFormatter機能を使用して、コードの自動フォーマットを行います。コミット前に自動実行するか、CIでチェックします。
*   **設定ファイル:** プロジェクトルートに `pyproject.toml` または `ruff.toml` を配置し、Ruffのルールセット（例: `select = ["E", "F", "W", "I", "UP", "PL"]`）、行長制限（例: 88文字または99文字）、対象外ファイルなどを設定します。

### **静的解析 (Static Analysis): Ruff**
*   **ルールセット:** Ruffのデフォルトルールセットに加え、プロジェクトの特性に応じて有用な追加ルール（例: `flake8-bugbear`, `flake8-comprehensions` などRuffが内包するルール）を有効化します。
*   **CI連携:** CI/CDパイプラインの初期段階でRuffによる静的解析を実行し、警告やエラーが検出された場合はビルドを失敗させるか、開発者に通知します。

### **CI/CD: GitHub Actions**
*   **ワークフロー:** `.github/workflows` ディレクトリにYAMLファイルでワークフローを定義します。
    *   **プッシュ/プルリクエスト時:**
        *   `develop` ブランチへのプッシュ時、および `main` / `develop` ブランチへのプルリクエスト時にトリガー。
        *   自動テスト実行: `unittest` を使用したテストスイートの実行。
        *   静的解析: Ruffによるコード解析とフォーマットチェック。
        *   カバレッジレポート: `coverage.py` を実行し、結果をCodecovなどのサービスにアップロードするか、アーティファクトとして保存。
    *   **リリース時 (オプション):**
        *   `main` ブランチへのマージ後、またはタグ作成時にトリガー。
        *   ドキュメント生成・デプロイ: Sphinxを使用してHTMLドキュメントをビルドし、GitHub Pagesに自動デプロイ。
        *   ゲームビルド: Pyxelのゲームを配布可能な形式（例: 実行可能ファイル、Web版の場合はHTML/JS）にパッケージングするスクリプトを実行し、リリースアーティファクトとしてGitHub Releasesにアップロード。（Pyxel自体に直接的なビルド機能は少ないため、PyInstallerやNuitka、またはWeb向け変換ツールなどを検討）

### **コードレビュー (Code Review): GitHub Pull Requests**
*   **必須プロセス:** `develop` ブランチや `main` ブランチへのすべてのマージは、Pull Request (PR) を介して行います。
*   **レビュー担当:** 各PRに対して、最低1名以上の他の開発者によるレビューと承認を必須とします。
*   **CIチェック連携:** PRが作成または更新されるたびに、上記のCIワークフロー（テスト、静的解析、フォーマットチェック）が自動的に実行され、すべてのチェックが成功することをマージの必須条件とします。

## 8. その他 (Miscellaneous)

### **依存関係管理 (Dependency Management):**
プロジェクトの Python パッケージ依存関係は `pyproject.toml` ファイルに記録することを推奨します (PEP 621 準拠)。
パッケージのインストール、仮想環境の作成・管理には、高速な Python パッケージインストーラーである `uv` の使用を推奨します。`uv` は `pip` や `venv` の代替として機能し、開発ワークフローの効率化に貢献します。
例えば、`uv venv` で仮想環境を作成し、`uv pip install -r requirements.txt` や `uv pip install <package>` (または `pyproject.toml` があれば `uv pip sync`) のようにして依存関係を管理します。
